# Technical Documentation: JHU Course Evaluation Analyzer

## 1. Introduction

This document provides a detailed technical breakdown of the JHU Course Evaluation Analyzer application, as described in `GEMINI.md`. It is intended for engineers who require a deep understanding of the system's architecture, data flow, and implementation details.

## 2. System Architecture

The application employs a modern, decoupled, full-stack architecture consisting of a React Single-Page Application (SPA) frontend and a Python/Flask backend API. This architecture separates concerns, with the backend managing data persistence and external data acquisition (scraping), while the frontend handles all user interaction, data processing, and presentation logic. The entire system is configured as a monorepo and deployed on the Vercel platform.

### 2.1. Architectural Components

-   **Frontend (React SPA)**: The primary interface for users. It is responsible for fetching raw data from the backend and performing all subsequent data analysis, filtering, and statistical calculations client-side via JavaScript.
-   **Backend (Python/Flask API)**: A stateless API that serves as a data gateway. Its sole responsibilities are to interact with the database and to scrape new course evaluation data from the JHU website on demand.
-   **Database (PostgreSQL on Supabase)**: The persistence layer for all course and metadata.
-   **Deployment (Vercel)**: The hosting platform that builds, deploys, and routes requests for both the frontend and backend services.

## 3. Data Model

The data is stored in a PostgreSQL database hosted by Supabase. The schema is designed to separate slowly changing course metadata from the more frequently updated raw evaluation data.

### 3.1. `course_metadata` Table

This table acts as a central registry for every course series known to the system.

| Column                            | Type      | Description                                                                                                                                                                                          |
| --------------------------------- | --------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `course_code`                     | `TEXT`    | **Primary Key**. The unique identifier for a course (e.g., `AS.180.101`). This represents the course as an entity, independent of any specific semester or instructor.                               |
| `last_period_gathered`            | `TEXT`    | The most recent academic period (e.g., "FA24") for which evaluation data was successfully scraped and stored. This is critical for the backend's logic to determine if the data is stale.             |
| `last_period_failed`              | `BOOLEAN` | A flag indicating if the most recent attempt to scrape a *new* academic period failed. This can be used for targeted error reporting or retry mechanisms.                                             |
| `relevant_periods`                | `JSONB`   | A JSON array of strings, where each string is an academic period for which this course has available evaluation data. Using `JSONB` allows for efficient querying of these periods.                  |
| `last_scrape_during_grace_period` | `DATE`    | A timestamp indicating the last time a scrape was attempted during the "grace period" (the time when new evaluations are being released but may not be complete). This prevents excessive scraping. |
| `created_at`                      | `TIMESTAMPTZ` | The timestamp when the course record was first created.                                                                                                                                              |
| `updated_at`                      | `TIMESTAMPTZ` | The timestamp of the last modification to this record. Automatically updated by the `trigger_set_timestamp` trigger.                                                                                 |

### 3.2. `courses` Table

This table stores the raw, unstructured evaluation data for each specific instance of a course.

| Column         | Type      | Description                                                                                                                                                           |
| -------------- | --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `instance_key` | `TEXT`    | **Primary Key**. A unique identifier for a specific course offering, typically a combination of the course code and the academic period (e.g., `AS.180.101_FA23`).       |
| `course_code`  | `TEXT`    | **Foreign Key** referencing `course_metadata.course_code`. This links the specific instance back to its parent course entity.                                            |
| `data`         | `JSONB`   | The complete, raw JSON payload scraped from the JHU evaluation website for this specific course instance. Storing it as `JSONB` allows for potential server-side queries into the data structure if needed in the future. |
| `created_at`   | `TIMESTAMPTZ` | The timestamp when the course instance record was created.                                                                                                            |
| `updated_at`   | `TIMESTAMPTZ` | The timestamp of the last modification. Automatically updated by the `trigger_set_timestamp` trigger.                                                                 |

### 3.3. Database Trigger: `trigger_set_timestamp`

A PostgreSQL trigger is configured to execute a function (`set_timestamp()`) before any `UPDATE` operation on both tables. This function automatically sets the `updated_at` column to the current transaction's timestamp, ensuring data modification times are always accurate without requiring application-level logic.

## 4. Backend (Python/Flask)

The backend is a lightweight Flask application whose responsibilities are narrowly defined.

-   **Primary Function**: It acts as a data service layer, abstracting the database and the scraping process from the frontend client.
-   **`CourseGroupingService`**: This service contains logic to identify cross-listed courses. The documentation notes it uses an "embedded configuration," which implies that the course groupings are defined in a local file within the backend's source code (e.g., a Python dictionary or a JSON file) rather than being stored in the database. This makes the grouping logic static and requires a code deployment to update.
-   **CORS Configuration**: The application uses the `Flask-Cors` extension to handle Cross-Origin Resource Sharing. It is configured to allow requests from:
    1.  The production frontend URL.
    2.  `localhost` (for local development).
    3.  A regular expression (`r"https://jhu-course-evaluation-analyzer-.*\.vercel\.app"`) to dynamically whitelist Vercel's preview deployment URLs.

## 5. Frontend (React)

The frontend is a complex SPA that contains the majority of the application's business logic.

-   **Client-Side Analysis**: The most significant architectural decision is that the frontend performs all data processing. The backend provides raw, aggregated data, and the frontend's `src/utils/analysisEngine.js` is responsible for:
    -   **Filtering**: Slicing the data by criteria such as academic year, instructor, etc.
    -   **Separation**: Grouping data for display (e.g., separating different sections of a course).
    -   **Statistical Calculations**: Computing all metrics, averages, and other statistics shown to the user.
-   **State Management**: While not explicitly stated, a standard React state management solution (e.g., Context API, Redux) is likely used to manage the application's state, including the raw data, filter settings, and computed results.
-   **Search History**: The `SearchHistory` component leverages the browser's `localStorage` via the `storageUtils.js` utility. This provides a simple, persistent, client-side storage mechanism for recent search queries.

## 6. Deployment and Configuration (Vercel)

The application is deployed as a monorepo on Vercel, orchestrated by the `vercel.json` configuration file.

-   **Build Process**: Vercel executes a two-stage build:
    1.  **Backend**: It uses the `@vercel/python` builder to install dependencies from `backend/requirements.txt` and configure the Flask application (`backend/app.py`) as a serverless function.
    2.  **Frontend**: It uses the `@vercel/static-build` builder, which executes the `build` script defined in `frontend/package.json` (typically `react-scripts build`). The output is a set of static files in the `frontend/build` directory.
-   **Routing Rules**: The `vercel.json` file defines how incoming requests are routed:
    -   `"/api/(.*)"`: All requests to this path are rewritten and proxied to the backend Python function.
    -   Static file paths (`/static/*`, `/favicon.ico`, etc.): These are served directly from the frontend's build output directory.
    -   **SPA Fallback**: All other requests are routed to serve `frontend/build/index.html`. This is a standard pattern for SPAs, allowing client-side routing libraries (like React Router) to handle the URL and render the appropriate view.
-   **Environment Variables**: The entire system relies on a single environment variable, `DATABASE_URL`, which must be set in the Vercel project settings. This variable contains the full connection string for the Supabase PostgreSQL database.

-   `POST` | `/api/analyze/<course_code>`           | This is the primary data-retrieval endpoint for the frontend. It aggregates and returns the complete raw evaluation data for the requested course *and* all other courses in its group (as determined by the `CourseGroupingService`). The response is a `raw_data` object containing all relevant course instances. Crucially, no analysis is performed by the backend; the raw data is sent directly to the client for processing. |
